# 筑览 ZhuLan — Atlas of Ancient Chinese Architecture
# 4C Competition 2026 — Category 6: AI + Information Visualization Design

## Project Overview
An interactive web application visualizing China's pre-1911 architectural achievements.
Users can explore structures on a map, filter by type and dynasty, and view data charts
showing architectural trends across Chinese history.

The dataset combines hand-curated "featured" entries (with rich descriptions and
AI-generated illustrations) with a larger set of structures sourced from Wikidata,
giving the app 50–100+ data points for meaningful visualizations.

Built for the 2026 (19th) Chinese Collegiate Computing Competition (4C).
Competition category: AI + Information Visualization Design (大类6)
Sub-tracks: Interactive Information Design (交互信息设计) + Data Visualization (数据可视化)

## Data Architecture

### Two-tier data model

**Tier 1 — Curated entries** (`/src/data/curated.json`)
- Hand-researched featured structures with full descriptions
- Include Chinese text, architect info, key features, AI-generated images
- Source field: `"source": "curated"`
- These are the "hero" items shown with expanded detail cards
- Target: 10–20 entries across all four types

**Tier 2 — Wikidata entries** (fetched by `/scripts/fetch-wikidata.ts`)
- Bulk-fetched from Wikidata SPARQL endpoint at build time
- Include: name (en + zh), coordinates, inception year, type, Wikimedia image URL
- Source field: `"source": "wikidata"`
- These fill out the map and charts with 50–100+ data points
- Shown with compact markers and basic info cards

**Merged output** (`/src/data/structures.json`)
- Generated by `npm run fetch-data` — DO NOT EDIT THIS FILE DIRECTLY
- The fetch script merges curated.json + Wikidata results, deduplicates by coordinates
- Curated entries always take priority over Wikidata duplicates

### Wikidata SPARQL query strategy
The fetch script queries `https://query.wikidata.org/sparql` for items that:
- Are located in China (P17 = Q148)
- Are instances/subclasses of: palace (Q16560), bridge (Q12280), residential building (Q11755880), or government building (Q16831714)
- Have coordinate location (P625)
- Have inception date (P571) before 1911
- Retrieve: label (en + zh), description, coordinates, inception, image (P18)

### Data schema
See `/src/types/index.ts` for full TypeScript interfaces.

Key fields for all structures:
- `id` — unique slug (e.g., "forbidden-city" or "wikidata-Q215380")
- `name` — English name
- `nameChinese` — Chinese name (中文名)
- `type` — one of: `"palace"` | `"residence"` | `"government"` | `"bridge"`
- `dynasty` — dynasty name in English (must match dynasties.json), or `"Unknown"`
- `yearBuilt` — approximate year (negative = BCE)
- `coordinates` — `[latitude, longitude]`
- `source` — `"curated"` | `"wikidata"` | `"ai-identified"` (temporary, from Scan & Discover)
- `status` — `"complete"` | `"ruin"` | `"reconstructed"`
- `significance` — `"national"` | `"regional"` | `"local"`

Additional fields for curated entries only:
- `description` / `descriptionChinese` — detailed text
- `architect` — builder/designer
- `keyFeatures` — array of notable characteristics
- `architecturalStyle` — style classification
- `materials` — construction materials
- `image` — path to AI-generated illustration
- `imageCredit` — AI tool used for image generation
- `sources` — academic/historical sources
- `aiToolsUsed` — which approved AI tools helped research

Additional fields for Wikidata entries:
- `wikidataId` — Wikidata Q-number (e.g., "Q215380")
- `wikimediaImage` — URL to Wikimedia Commons image (if available)
- `wikipediaUrl` — link to Wikipedia article (if available)

## Scan & Discover Feature (筑览识图)

### Overview
The signature AI-powered feature of ZhuLan. A user photographs or uploads an image of
a Chinese historical structure, and the app uses Tongyi Qwen-VL to identify it, fly to
its location on the map, and display rich historical context.

This feature is the primary demonstration of "AI tool mastery" for competition judging.

### User Flow
```
1. User taps the scan button (floating action button, bottom-right of map)
2. Modal opens with two options:
   a. "Take Photo" → opens device camera (mobile) or webcam (desktop)
   b. "Upload Image" → file picker for jpg/png
3. Image is captured/selected → show preview with "Identify" button
4. User taps "Identify" → loading state with animation
5. Image is sent to Qwen-VL via Dashscope API
6. API returns structured JSON with identification results
7. App processes the result:
   a. If structure exists in our dataset → fly to it on map, open its info panel
   b. If structure is recognized but NOT in dataset → create temporary pin on map,
      show AI-generated info card with historical facts
   c. If structure is not recognized → show friendly "not recognized" message
      with suggestion to try another angle or a different structure
8. Result card shows: name (en/zh), dynasty, type, historical facts, confidence
```

### Dashscope API Integration

**Service file:** `src/services/qwenVL.ts`

**API endpoint:** `https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions`

**Model:** `qwen-vl-max` (best accuracy for landmark recognition)

**Authentication:** API key via `VITE_DASHSCOPE_KEY` environment variable

**Request format** (OpenAI-compatible):
```typescript
const response = await fetch('https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${import.meta.env.VITE_DASHSCOPE_KEY}`,
  },
  body: JSON.stringify({
    model: 'qwen-vl-max',
    messages: [{
      role: 'user',
      content: [
        {
          type: 'image_url',
          image_url: { url: `data:image/jpeg;base64,${base64Image}` }
        },
        {
          type: 'text',
          text: IDENTIFY_PROMPT  // see prompt below
        }
      ]
    }],
  }),
});
```

**System prompt for identification:**
```
You are an expert on ancient Chinese architecture (pre-1911). Analyze this photo and
identify the structure. You MUST respond with ONLY valid JSON, no markdown, no preamble.

If you can identify the structure, respond with:
{
  "identified": true,
  "confidence": "high" | "medium" | "low",
  "name": "English name",
  "nameChinese": "中文名",
  "type": "palace" | "bridge" | "residence" | "government",
  "dynasty": "Dynasty name in English",
  "estimatedYear": 1420,
  "coordinates": [latitude, longitude],
  "province": "Modern province name",
  "city": "City name",
  "historicalFacts": [
    "Fact 1 in English",
    "Fact 2 in English"
  ],
  "historicalFactsChinese": [
    "事实1",
    "事实2"
  ],
  "architecturalStyle": "Style description",
  "significance": "Why this structure matters"
}

If you cannot identify it, respond with:
{
  "identified": false,
  "reason": "Brief explanation of why identification failed",
  "suggestion": "Helpful suggestion for the user"
}

Important rules:
- Only identify structures that are pre-1911 Chinese architecture
- Only classify as: palace, bridge, residence, or government building
- If the photo shows a temple, pagoda, or modern building, set identified to false
- Coordinates must be accurate — do not guess if unsure
- Historical facts should be verifiable
```

### Matching against the dataset
After receiving the API response, the app should attempt to match the identified structure
against the existing dataset (structures.json):

1. **Name match:** Compare `name` or `nameChinese` against all entries (fuzzy match)
2. **Coordinate match:** If name doesn't match, check if coordinates are within ~0.05°
   (~5km) of any existing entry
3. **If matched:** Fly to the existing entry, highlight it, open its full info panel
4. **If not matched:** Create a temporary `Structure` object from the AI response with
   `source: "ai-identified"`, add it as a temporary marker on the map, show the
   AI-generated info card

### Component structure

**ScanButton.tsx** — Floating action button
- Positioned bottom-right of map view, above zoom controls
- Camera icon (use a Lucide icon or simple SVG)
- Subtle pulse animation to draw attention
- On click → opens CameraCapture modal

**CameraCapture.tsx** — Photo capture modal
- Full-screen overlay on mobile, centered modal on desktop
- Two-tab interface: "Camera" and "Upload"
- Camera tab: uses `navigator.mediaDevices.getUserMedia` for live preview
  - Capture button snaps a photo from the video stream
  - Must handle permission denial gracefully
- Upload tab: drag-and-drop zone + file input, accept `image/jpeg,image/png`
- After capture/upload: show image preview + "Identify Structure" button
- Handle image compression: resize to max 1024px on longest side before sending
  (reduces API cost and latency, Qwen-VL doesn't need full-res)

**IdentifyResult.tsx** — Result display card
- Slides up from bottom (mobile) or appears as panel (desktop)
- Shows: structure image (user's photo), name (en + zh), dynasty badge, type icon
- Historical facts as a short scrollable list
- "View on Map" button → flies to location
- "Confidence" indicator: high = green, medium = amber, low = red
- If not identified: friendly message + retry button

### Error handling
- **API key missing:** Show setup instructions, not a cryptic error
- **API rate limit / 429:** "Too many requests, please try again in a moment"
- **Network error:** "Unable to connect. Check your internet connection."
- **API returns non-JSON:** Retry once, then show generic error
- **Image too large:** Compress client-side before sending (max ~500KB after compression)
- **Camera permission denied:** Fall back to upload-only mode with explanation
- **Non-architecture photo:** The prompt handles this — show "not recognized" result

### Cost considerations
- Qwen-VL-Max: ~$0.40 per million input tokens (images are ~1,000 tokens each)
- Estimated cost per identification: ~$0.001 (less than a tenth of a cent)
- For competition demo: 50 identifications = ~$0.05 total
- Set a spending alert on the Dashscope dashboard just in case

### Privacy
- Images are sent to Alibaba Cloud's Dashscope API for processing
- No images are stored server-side (stateless API call)
- Show a brief privacy notice on first use: "Your photo will be sent to Tongyi AI
  for identification. No images are stored."
- Do not send images automatically — always require user to tap "Identify"

## Map Configuration

### Tianditu (天地图) Setup
Tianditu is China's official government map service. Using it satisfies competition rule 5
regarding map compliance with PRC standards.

- **API Key:** Register at https://console.tianditu.gov.cn/ (free, personal developer)
- **Tile layers:** Two layers overlaid in Leaflet:
  1. Vector base: `vec_w` (矢量底图)
  2. Chinese annotation: `cva_w` (矢量注记)
- **Projection:** Web Mercator (EPSG:3857) — use the `_w` suffix tiles
- **Tile URL pattern:**
  ```
  http://t{s}.tianditu.gov.cn/{layer}/wmts?SERVICE=WMTS&REQUEST=GetTile&VERSION=1.0.0&LAYER={layerCode}&STYLE=default&TILEMATRIXSET=w&FORMAT=tiles&TILEMATRIX={z}&TILEROW={y}&TILECOL={x}&tk={API_KEY}
  ```
  Where `{s}` is 0–7 for load balancing.
- **Subdomains:** `['0','1','2','3','4','5','6','7']` for `t0`–`t7`

### Map defaults
- Default center: `[35.8617, 104.1954]` (geographic center of China)
- Default zoom: `5`
- Min zoom: `3`, Max zoom: `18`

### Marker colors by structure type
- palace → red `#DC2626`
- residence → amber `#D97706`
- government → blue `#2563EB`
- bridge → teal `#0D9488`

### Important: API key handling
Store API keys in a `.env` file (see `env.example` for template):
```
VITE_TIANDITU_KEY=your_tianditu_key_here
VITE_DASHSCOPE_KEY=your_dashscope_key_here
```
Access in code via `import.meta.env.VITE_TIANDITU_KEY` and `import.meta.env.VITE_DASHSCOPE_KEY`.
Add `.env` to `.gitignore`. For GitHub Pages deployment, set keys as GitHub Actions
secrets or hardcode in the deploy build (Tianditu keys are domain-restricted;
Dashscope keys should have a spending cap set on the Alibaba Cloud console).

**Note:** The Dashscope API key is exposed in the client bundle. This is acceptable for
a competition project. For production, you'd proxy through a backend. Set a spending
alert (~$1) on the Dashscope dashboard to avoid surprises.

## Architecture Rules
- **All data lives in /src/data/*.json** — never hardcode content in components
- One component per file. File name = component name (PascalCase)
- Shared types go in `/src/types/index.ts` — always import from there
- No inline styles — use Tailwind utility classes only
- All map markers must use the `Structure` type from `/src/types/index.ts`
- D3 charts must be wrapped in React components using `useRef` + `useEffect` pattern
- All text visible to users must support both English and Chinese (use `name` and `nameChinese` fields)
- Curated vs Wikidata vs AI-identified entries should render differently:
  - Curated: full detail panel with description, features, image
  - Wikidata: compact info card with name, type, date, link to Wikipedia
  - AI-identified: temporary marker with AI-generated facts, "confidence" badge, no persistence

## Coding Conventions
- **TypeScript strictly** — no `any` types. Define proper interfaces in `/src/types/`
- **Functional components only** — no class components
- **camelCase** for variables and functions
- **PascalCase** for components, types, and interfaces
- **kebab-case** for CSS class names (when not using Tailwind)
- **ES modules** — never use CommonJS `require()`
- Prefer `const` over `let`. Never use `var`
- All async operations must handle errors with try/catch
- Keep components under 150 lines — split if longer

This app deploys to GitHub Pages. It is a fully static build — no server.

```typescript
export default defineConfig({
  base: '/zhulan/',
  plugins: [react()],
```

### package.json deploy script
```json
{

    "gh-pages": "^6.0.0"
```

### React Router — use HashRouter not BrowserRouter
GitHub Pages does not support client-side routing fallbacks.
```typescript

// URLs: /zhulan/#/timeline
```
Never use BrowserRouter — it breaks on refresh on GitHub Pages.

## Common Mistakes to Avoid
- Do not use `react-leaflet` SSR features — this is a Vite CSR app
- Do not mutate D3 selections outside useEffect cleanup
- Do not import all of D3 — use named imports only (tree-shaking)
- Do not use percentage widths on Leaflet container — use explicit pixel or vh heights
- Do not edit `/src/data/structures.json` directly — edit `curated.json` instead
- Do not use OpenStreetMap tiles — use Tianditu for PRC map compliance
- Tianditu uses `{x}`, `{y}`, `{z}` in URLs — same as Leaflet expects, but the
  parameter names in the WMTS URL are `TILECOL`, `TILEROW`, `TILEMATRIX`
- The Wikidata fetch script must handle: missing labels, missing dates, coordinate
  format conversion (Wikidata uses `Point(lng lat)`, we need `[lat, lng]`)
- Dashscope API returns may include markdown fencing around JSON — strip ```json and ```
  before parsing
- Always compress images before sending to Qwen-VL — max 1024px longest side, JPEG
  quality 0.8. Full-res photos waste tokens and add latency
- Camera API (`getUserMedia`) requires HTTPS — works on localhost but NOT on plain HTTP.
  GitHub Pages serves HTTPS by default, so this is fine in production
- Do not auto-send images on capture — always let user confirm with "Identify" button
  (privacy + cost control)
- Qwen-VL may hallucinate coordinates for lesser-known structures — always show
  confidence level and let user know AI results may not be 100% accurate
